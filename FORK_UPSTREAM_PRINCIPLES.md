# OpenMAIC 二开与 Upstream 同步原则

本文档是本项目长期有效的开发原则，目标是同时满足三件事：

1. 支持业务二开和商业化落地
2. 尽量降低与上游 `upstream` 的冲突成本
3. 保持后续持续吸收上游更新的能力

除非有明确、可记录的业务理由，否则后续开发默认遵守本文档。

## 1. 总原则

### 1.1 薄分叉优先

本项目应保持 `thin fork`，即：

- 能新增文件解决，就不要直接改上游原文件
- 能通过配置解决，就不要通过改核心逻辑解决
- 能通过适配层解决，就不要直接侵入底层实现
- 能通过独立服务解决，就不要把业务耦合进上游核心

### 1.2 配置优先于硬编码

所有部署相关、环境相关、入口路径相关、第三方服务相关的差异，优先做成配置。

典型包括：

- `basePath`
- 域名与端口
- 模型服务商与 API Key
- 功能开关
- 外部服务地址

原则是：换环境时优先改配置，不优先改代码。

### 1.2.1 basePath 必须配置化

`basePath` 属于典型的环境差异，不允许写死在业务代码里。

统一约定：

- 根路径部署时，不设置 `NEXT_PUBLIC_BASE_PATH`
- 子路径部署时，通过 `NEXT_PUBLIC_BASE_PATH` 指定，例如 `/maic`

开发要求：

- 不要把 `/maic`、`/demo`、`/xxx` 这类环境路径直接写进代码
- 新增页面、静态资源、前端请求路径时，必须兼容“根路径部署”和“子路径部署”两种模式
- 如果某段实现只能在固定子路径下工作，应视为缺陷，而不是可接受方案

判断标准：

- 换一个部署路径时，应该只需要改环境变量并重新构建部署
- 不应该因为换路径而再次修改源码

### 1.3 增量改造优先于重写

如果一个需求可以通过“新增一层”完成，就不要重写原有流程。

优先顺序：

1. 新增页面 / 新增组件
2. 新增 API route / 新增 service
3. 新增 adapter / wrapper / facade
4. 最后才是改原有核心文件

### 1.4 保持上游可合并性

任何改动都要先问一句：

“如果上游三个月后改了同一块，这个改法是否还容易合并？”

如果答案是否定的，应优先调整方案。

## 2. 高冲突区定义

以下类型的内容默认视为高冲突区，应少改、慎改：

- 框架入口与全局配置
- 全局布局、全局样式、共享主题
- 核心路由与主流程入口
- 通用状态管理与全局 store
- 底层通用工具、编排核心、通用协议层
- 锁文件、构建链、包管理器配置
- vendored / forked 第三方代码

对高冲突区的要求：

- 没有明确理由不要改
- 必须改时，改动面要尽量小
- 改完必须补充“为什么不能用外围扩展方式解决”的说明

## 3. 默认推荐的二开方式

### 3.1 用新增目录承载业务定制

自有业务逻辑优先放在新增模块中，而不是塞进上游已有模块。

建议优先采用：

- 新增业务页面
- 新增业务组件
- 新增业务 API
- 新增服务封装层
- 新增集成层

### 3.2 用适配层隔离外部系统

支付、租户、权限、内部系统对接、统计、审计、企业工作流等自有逻辑，优先拆到适配层或独立服务。

不建议直接把这类逻辑耦合进上游核心课堂生成流程。

### 3.3 用环境变量驱动部署差异

部署路径、反向代理、模型供应商、开关项等，必须通过环境变量或配置文件控制。

禁止为了某个环境直接写死：

- 域名
- 子路径
- 端口
- 私有 API 地址
- 某个环境专用逻辑

当前项目中，`basePath` 的唯一合法来源应是环境配置，而不是业务代码常量。

### 3.4 保留上游默认行为

如果新增企业化能力，优先做到：

- 不影响默认开源体验
- 不破坏上游原始默认值
- 新能力通过开关启用

## 4. 明确禁止的做法

除非有强业务理由并经过记录，禁止以下做法：

- 为了“代码看起来更整齐”而批量移动上游文件
- 没有业务价值的大规模重命名
- 无必要的大规模格式化、import 排序、换行风格重排
- 在一个提交里同时做功能改动、重构、格式化
- 直接改第三方包源码而不做隔离说明
- 把临时脚本、调试逻辑、环境特判长期留在核心流程
- 在 `main` 上直接做实验性开发

这些做法会显著放大未来与上游同步时的冲突面。

## 5. 分支与同步策略

### 5.1 远程命名约定

- `origin`：自己的 fork
- `upstream`：原项目仓库

### 5.2 本地分支建议

- `main`：自己的稳定主线
- `feature/*`：功能开发分支
- `integration/upstream-*`：吸收上游更新的集成分支
- `hotfix/*`：线上修复

### 5.3 同步上游的标准方式

不要把上游更新直接粗暴合进正在开发的功能分支。

标准流程应是：

1. `git fetch upstream`
2. 从自己的 `main` 切出集成分支
3. 在集成分支合并 `upstream/main`
4. 解决冲突并跑测试
5. 集成分支验证通过后，再合回自己的 `main`
6. 各个功能分支再从新的 `main` 同步

### 5.4 小步同步，不要长期脱节

原则上应频繁同步上游，而不是一次性跨很多版本同步。

推荐节奏：

- 至少每 1 到 2 周检查一次上游更新
- 遇到安全修复、依赖修复、关键 bug 修复时优先同步

## 6. 冲突处理优先级

当自己的改动与上游冲突时，默认优先级如下：

1. 优先保留上游 bugfix、安全修复、兼容性修复
2. 若业务能力必须保留，优先改造成外围扩展
3. 如果必须继续改核心，务必缩小改动边界
4. 冲突解决后，补充验证和说明

不要因为“先前已经改过”就默认压掉上游更新。

## 7. 开发前检查清单

开始实现前先自检：

- 这个需求能否通过新增模块完成
- 这个差异能否通过配置完成
- 是否会修改高冲突区
- 是否会影响上游默认行为
- 未来上游更新到来时，这个改动是否容易合并

如果其中任一项答案明显不理想，应先改方案再写代码。

## 8. 提交要求

提交必须尽量保持“单一目的”。

推荐做法：

- 一个提交只做一类事情
- 功能改动与重构分开
- 路径兼容、部署兼容、业务功能分开提交
- 提交说明写清楚“为什么这样做”

如果修改了高冲突区，提交说明中必须额外写清：

- 为什么必须改这里
- 是否尝试过外围扩展方案
- 后续与 upstream 合并时要注意什么

## 9. 测试与验收要求

凡是涉及以下内容的改动，必须至少做一次针对性验证：

- 路由
- API
- 静态资源路径
- 环境变量
- 构建流程
- 反向代理 / 子路径部署
- 与上游核心流程交互的业务扩展

原则是：改动越靠近基础设施，验证越不能省。

## 10. 文档要求

以下情形必须补文档：

- 新增环境变量
- 新增部署约定
- 新增外部依赖
- 修改同步策略
- 修改高冲突区
- 引入必须长期遵守的开发约束

文档优先写“原则、边界、原因”，少写容易过期的临时细节。

## 11. 许可证与归属

本项目基于上游开源项目二开，必须持续保留原有许可证与版权归属信息。

如果后续对外提供网络服务并运行修改版本，应同时关注：

- 上游许可证义务
- 修改版本源码可获得性义务
- 第三方依赖许可证兼容性

业务目标不能成为忽略许可证约束的理由。

## 12. 一句话决策规则

如果某个改动会让未来“更难同步 upstream”，那它默认不是第一选择。

优先选择：

- 更容易配置
- 更容易隔离
- 更容易测试
- 更容易合并上游

而不是一时最快、但长期最难维护的方案。
