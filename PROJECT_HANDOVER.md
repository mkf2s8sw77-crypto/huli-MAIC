# huli-MAIC 项目交接文档

本文档面向后续接手本项目的新工程师，目标不是记录某一次发布的临时细节，而是提供一份长期有效、相对稳定的宏观交接说明。

建议把它当作“进入项目后的第一份必读文档”。

---

## 1. 项目是什么

本项目是基于上游开源项目 `THU-MAIC/OpenMAIC` 持续二开的业务化版本。

可以把它理解成：

- **上游项目**：提供“多智能体互动课堂”这一开源产品能力的通用底座
- **当前项目**：在保留持续吸收上游更新能力的前提下，完成品牌化、部署化、商用化和模型/媒体能力扩展的 fork

当前项目不是一次性魔改分支，而是一个**长期维护的 fork 仓库**。

这意味着后续开发必须同时兼顾两件事：

1. 满足自身业务演进
2. 保持对上游 `upstream` 的可同步性

---

## 2. 仓库关系与 Git 约定

当前默认约定是：

- `origin`：自己的 fork 仓库
- `upstream`：`THU-MAIC/OpenMAIC`

理解方式：

- 所有业务化修改都在自己的 fork 上持续积累
- 上游一旦有新功能、bugfix、安全修复，可以定期同步进来

本项目已经不是“下载下来的源码快照”，而是正式 Git 仓库。

### 长期维护原则

- 不要把本项目当成一次性拷贝
- 不要脱离 `upstream` 长期野蛮发展
- 任何明显增加未来合并冲突面的改动，都要谨慎评估

关于更细的同步原则，请优先阅读：

- `FORK_UPSTREAM_PRINCIPLES.md`

---

## 3. 许可证与对外提供服务

本项目继续继承上游的开源许可证约束。  
这不是“参考了一个开源项目”，而是**基于开源项目持续二开并对外提供服务**。

新接手工程师必须知道三点：

1. 许可证问题不是法务附属事项，而是工程约束的一部分
2. 对外提供网络服务时，源码可获得性、许可证展示、开源说明入口都必须持续满足
3. 前端品牌化、界面改造、部署路径调整，不会消除原有开源义务

工程上已经做过的合规化动作包括：

- 站内保留源码/许可证/开源说明入口
- 站点文案明确“基于 OpenMAIC 二次开发”
- fork 仓库持续公开

后续工程师不要删除、弱化或绕过这些入口，除非明确经过新的合规评估。

## 4. 项目的核心技术形态

这是一个 **Next.js + React + TypeScript** 项目，但它不是普通的信息展示站，而是一个“课程生成 + 课堂播放 + 多智能体互动 + 导出”的复合系统。

从宏观上看，可以分成 4 层：

### 4.1 课程生成层

输入是：

- 用户需求文本
- 可选 PDF
- 可选 Web Search
- 可选多媒体生成能力

输出是：

- scene outlines
- slide / quiz / interactive / pbl scene 内容
- actions（讲解、讨论、白板、播放等）

这是项目最核心的“AI 课程生产流水线”。

### 4.2 课堂播放层

生成好的场景会进入课堂页，以“播放引擎 + 多智能体 + 聊天区 + 白板 + 圆桌区”的方式运行。

这不是简单地把一组 slide 渲染出来，而是带有：

- 讲解动作
- 多智能体讨论
- TTS / ASR
- 白板与高亮效果
- 交互式页面与 PBL

### 4.3 媒体能力层

本项目已经不是单纯依赖 LLM：

- LLM：课程内容、scene、agent、actions
- Image：图片生成
- TTS：语音合成
- ASR：语音识别
- PDF：文档解析
- Web Search：联网资料补充

这些能力是“应用层编排多个模型服务”的结果，而不是单模型解决所有事。

### 4.4 导出层

当前支持导出：

- PPTX
- 教学资源包（PPTX + 交互页面等）

导出已经不再是固定横版逻辑，当前已支持多比例。

## 5. 当前二开的长期方向

当前 fork 的长期方向不是“完全脱离 OpenMAIC 自己重写”，而是：

- 保留上游课堂底座
- 逐步加入适合业务使用的品牌化、部署化、媒体化和商用化能力
- 尽量把差异做成可配置、可复用、可维护的结构

简单说，目标不是变成一个不可合并的私有分叉，而是保持：

- 可持续二开
- 可持续吸收 upstream
- 可对外提供服务

## 6. 目前已经形成的关键工程约束

这些内容属于本项目的“长期规则”，不是临时习惯。

### 6.1 配置优先，不要写死环境差异

所有环境差异优先走配置，而不是写死在代码里。

尤其是：

- `basePath`
- 域名
- 端口
- 模型 provider
- API Key
- 功能开关
- 是否隐藏某些前端入口

如果换环境需要改源码而不是改配置，通常说明设计不够好。

### 6.2 前端不负责保存生产密钥

正式环境、预发布环境、对外服务环境，默认都应由**服务端环境变量**提供模型密钥。

原则是：

- 正式 key 不存浏览器 localStorage
- 前端 UI 可展示、可只读、可受控
- 生产 key 的主配置源必须是服务端

### 6.3 面向外部用户的配置要收口

如果站点是对外服务，不要把“面向工程师的 provider 配置能力”直接暴露给终端用户随意修改。

目前已经采用的思路是：

- 服务端统一配置 provider
- 前端媒体配置可锁为只读
- 某些不对外开放的媒体能力入口可隐藏

这类策略后续应继续保持，而不是重新把生产配置入口开放给终端用户。

### 6.4 basePath 必须长期保持配置化

本项目已经明确按“根路径 / 子路径均可部署”的方向建设。

当前原则是：

- 根路径部署：不设 `NEXT_PUBLIC_BASE_PATH`
- 子路径部署：通过 `NEXT_PUBLIC_BASE_PATH` 设置，例如 `/maic`

任何新页面、新资源、新接口引用，都必须兼容 basePath。

### 6.5 比例不是导出层的临时选项，而是课程级配置

当前项目已开始把“课程比例”做成课程生成参数，而不是只在导出时临时切换。

支持的比例包括：

- `16:9`
- `4:3`
- `3:4`
- `9:16`

关键认知：

- 比例会影响课程生成 prompt
- 会影响课堂主画布
- 会影响缩略图
- 会影响 PPT 导出 layout

所以后续如果继续扩展比例，不要只改导出逻辑，必须按“生成 -> 播放 -> 导出”全链路思考。

### 6.6 头像资源必须做兜底

多智能体 agent 的头像不是完全可信输入。

原因：

- LLM 可能返回不规范路径
- 历史数据可能存在旧格式
- 部署子路径可能影响资源引用

因此工程上已经形成一个明确原则：

- 头像路径必须统一归一化
- 必须存在 fallback
- 渲染层不要直接无条件相信 `agent.avatar`

后续扩展角色系统时，应继续沿用这个原则。

## 7. 本项目与上游最容易冲突的区域

新工程师必须特别注意以下目录/模块，因为它们是最容易和 upstream 冲突的区域：

- provider / settings / i18n 相关配置层
- generation pipeline
- audio / media provider 抽象层
- store 层
- 导出层
- 课堂主画布与布局层

这些区域之所以容易冲突，是因为：

- 你在做业务化扩展
- 上游也会持续增加 provider、能力和 UI

典型例子：

- 你新增 `MiniMax`，上游新增 `ElevenLabs`
- 你新增 `MiniMax Image`，上游新增 `Grok Image`
- 你新增媒体锁定逻辑，上游调整设置页结构

这类冲突不是偶发，而是长期会出现的常态。

因此对这些区域的开发要求是：

- 尽量小改
- 尽量聚焦
- 尽量保留扩展式结构
- 任何新 provider 都要按“类型 / 常量 / provider-config / settings / i18n / UI”整链路补齐

## 8. 新工程师必须理解的配置哲学

下面这些不是单个环境的偶然设置，而是本项目已经形成的配置哲学。

### 8.1 服务端环境变量是主配置源

`.env.example` 的职责是：

- 告诉后来者“有哪些长期有效的可配置项”
- 作为不同环境之间迁移的模板

`.env.local` 的职责是：

- 当前环境的真实私密配置

原则上：

- 提交模板，不提交密钥
- 维护 `.env.example`
- 不要把临时环境变量散落在代码注释和个人脚本里

### 8.2 可以锁的入口必须做成开关

类似下面这类“面向业务运营/终端用户”的差异，应优先做成前端开关，而不是分叉 UI：

- 媒体设置是否只读
- Video 入口是否展示
- 某些 provider 是否只给内部可见

也就是说，本项目应该尽量把“对外版”和“内部调试版”的差异收敛为配置，而不是维护两套页面。

### 8.3 默认值本身是产品决策

比如：

- 默认课程比例
- 默认 provider
- 默认语言
- 默认媒体开关

这些看起来像小配置，但实际上会影响用户体验、内容生成结果和导出效果。

不要随手改默认值；改默认值时要把它当产品决策看待。

## 9. 数据存储与运行时认知

本项目不是传统“服务端数据库驱动一切”的架构，存在明显的本地优先特征。

### 9.1 浏览器端有本地持久化

项目使用本地持久化保存：

- stage / scenes
- chat session
- outlines
- 生成媒体
- generated agents
- 多类 settings

因此调试时要有这个意识：

- 某些问题来自当前浏览器已有的本地状态
- 不是所有问题都来自服务端
- 同一套代码在“全新浏览器环境”和“已有历史数据的浏览器环境”表现可能不同

### 9.2 本地状态与服务端配置会叠加

服务端 provider 配置、浏览器历史设置、课程本地缓存，这三类状态会同时影响最终行为。

调试时通常需要先分清：

1. 是服务端 provider 配置问题
2. 是浏览器 local state 问题
3. 是某个历史 stage 数据问题

这类认知对排障非常关键。

## 10. 媒体与模型能力的长期扩展方式

如果后续继续扩展新的模型供应商，推荐按以下顺序理解和落地：

1. 类型层  
   先补 provider ID / 类型定义

2. 常量层  
   补 provider metadata、icon、默认模型、voice、能力说明

3. 服务端配置层  
   补环境变量映射 / server provider 解析

4. 实现层  
   新增 adapter / API 调用实现

5. 设置层  
   补 settings store 默认值和 server merge 逻辑

6. 界面层  
   补设置页、选择器、图标映射、只读/隐藏逻辑

7. 文案层  
   补 i18n

这个顺序很重要。  
如果只补其中一半，项目会进入“能调 API 但 UI 不认、或 UI 有入口但服务端不通”的半完成状态。

## 11. 比例功能的长期演进方式

今后如果要继续扩展比例能力，建议遵守以下原则：

### 11.1 比例是课程级能力，不是单页临时属性

不要把比例做成“某一页自己决定”。  
当前更合理的抽象是：

- 课程 / stage 级别决定主比例
- scene 默认沿用 stage 比例
- 导出沿用课程比例

### 11.2 比例变更必须考虑四层

后续任何比例功能改造，都至少要检查这四层：

1. 生成参数
2. 课堂渲染
3. 缩略图/导航
4. 导出 PPT

如果只改其中一层，最终体验一定不一致。

### 11.3 竖版不是把横版压窄

竖版课程要想好看，核心不是“压缩横版”，而是：

- prompt 要知道目标尺寸
- 布局要适配更长的纵向空间
- 图片/视频比例要重新选择
- 留白和字号要跟着调整

这类事情后续如果继续优化，重点应放在生成 prompt 和布局规则，而不是只放在 CSS。

## 12. 部署与运行的长期原则

这里不记录某一台机器的所有临时细节，而只保留长期有效原则。

### 12.1 子路径部署必须长期受支持

当前项目已经明确支持子路径部署。  
这不是临时 hack，而是正式部署能力的一部分。

### 12.2 部署版本与公开源码应尽量一致

如果站点对外运行的是某个版本，公开仓库中应尽量能找到对应版本源码。

工程上至少应做到：

- 部署前有明确 commit
- 必要时有 tag
- 部署版本与仓库状态能对应起来

### 12.3 预发布与生产不应靠手工差异维持

不同环境应该依靠：

- 环境变量
- 配置文件
- 部署参数

而不是靠工程师记忆中的“这台机器要手工改哪一行”。

如果某项差异需要靠口口相传，那它就应该被配置化或文档化。

### 12.4 `dev.huli.sh.cn` 是共享 host 网关，不是单项目入口

当前这台开发机上的预发布环境，已经收敛成“一个 host 级共享网关 + 多个 path 路由”的结构。

长期应这样理解：

- `dev.huli.sh.cn` 是这台机器上的**预发布共享入口**
- 本机 Nginx 共享网关文件是：
  - `/opt/homebrew/etc/nginx/servers/dev-huli-gateway.conf`
- 不同项目通过 path 挂载到同一个 host 下，例如：
  - `/maic/`
  - `/palliative-care/`
  - `/thyroid-rescue-game/`

这意味着：

- 不要再把 `dev` 环境理解成“每个项目一个独立 host 配置文件”
- 新增预发布服务时，默认是在共享 gateway 下新增 path 路由
- 共享 gateway 文件必须保持中性命名，不应再回退成某个业务项目专属名称

### 12.5 `dev` 预发布公网链路依赖现成 ingress 层

当前 `dev` 的公网访问不是项目自带能力，而是“本机发布层 + 现成 ingress 层”的组合。

宏观链路是：

1. 工程师在本机执行 `app-deploy`
2. `app-deploy` 将服务接入本机共享 gateway
3. 共享 gateway 通过 host 级 tunnel 暴露到 `dev.huli.sh.cn`
4. 外部用户通过 `https://dev.huli.sh.cn/<path>/` 访问

按当前环境，关键锚点是：

- 远端 Caddy：`dev.huli.sh.cn -> 127.0.0.1:19012`
- host 级 tunnel：`dev-huli-gateway-tunnel`
- tunnel 映射：远端 `19012 -> 本机 8080`
- 本机共享 gateway：`dev-huli-gateway.conf`

这里最重要的认知不是某个具体端口号，而是：

- `app-deploy` 负责本机服务接入共享 gateway
- 现成 ingress 层负责把这个 gateway 暴露到公网
- 对调用者来说，在“公网接入层已存在且正常”的前提下，执行 `app-deploy` 即可完成该环境的公网发布接入

### 12.6 `sys.huli.sh.cn` 不是这台机器的默认维护边界

必须明确区分：

- `dev.huli.sh.cn`：本机预发布环境
- `sys.huli.sh.cn`：生产环境，不应默认按本机 `dev` 链路推断

尤其要记住：

- `sys.huli.sh.cn` 的入口和 `19000` 端口，不属于这台开发机的默认维护范围
- 未确认责任边界前，不要在这台机器上擅自调整 `sys` 的反代、端口映射或 tunnel 假设

如果后续新工程师需要处理 `sys`，应该先确认：

- 生产机的本机 gateway 文件
- 生产机的 ingress 层
- 生产域名是否具备与预发布同构的发布前提

确认完这些前提后，才能安全地在生产环境复用 `app-deploy`

## 13. 新工程师接手后的第一优先事项

建议按这个顺序建立认知：

1. 先理解 fork / upstream 关系  
   看 `FORK_UPSTREAM_PRINCIPLES.md`

2. 再理解项目的四层结构

3. 再理解服务端配置优先的原则

顺序反过来会很容易把问题看成“单页 UI 问题”，但这个项目的复杂度远高于普通前端站点。

## 16. 本文档的定位

本文档不是：

- 发布记录
- 环境变量清单
- 临时部署脚本说明
- 单次故障排查记录

本文档是：

- 项目长期定位说明
- 架构与维护边界说明
- 新工程师进入项目后的高层地图

如果未来项目继续演进，本文档应该只在以下情况更新：

- 项目定位发生变化
- fork/upstream 策略发生变化
- 许可证/对外服务方式发生变化
- 配置哲学发生变化
- 生成 / 播放 / 导出 / 媒体的架构主线发生变化

也就是说，它应该是“低频更新、长期有效”的交接文档。
